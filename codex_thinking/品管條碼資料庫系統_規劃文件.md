# 品管條碼資料庫系統 — 規劃文件

> 版本：v1.0　|　目標環境：個人單機 / Python / SQLite
> 定位：承接「即時條碼辨識系統」解出的條碼數字，記錄 **NG / OK** 判定，並提供查詢、統計與匯出。

---

## 1. 系統目標與範圍

### 1.1 要做的事
- 接收**即時辨識系統**解碼出的條碼數字（EAN-13 等）。
- 由人工標記該件產品為 **OK（合格）** 或 **NG（不合格）**。
- 將每一筆判定寫入本機資料庫，永久保存。
- 提供查詢、歷史回溯、統計（不良率、缺陷趨勢）與匯出（CSV / Excel）。

### 1.2 明確「不做」的事（避免過度設計）
- 不做多人協作、不做帳號權限（單機個人）。
- 不做雲端、不做即時辨識（辨識端已獨立存在）。
- 第一版不記錄量測數值、不做 SPC（保留為未來擴充，見第 11 節）。

### 1.3 邊界
本系統只負責**「資料記錄與分析」**。條碼的擷取與解碼由既有的即時辨識系統處理，兩者透過一個共用的資料存取模組對接（見第 9 節）。

---

## 2. 核心設計原則（重要觀念）

### 2.1 ⚠️ 條碼是「品項識別」不是「單件識別」
EAN-13 條碼識別的是**商品種類**，同一款產品每一件貨上的條碼都相同。因此：

- 條碼**不可**作為檢驗記錄的主鍵，否則檢驗第二件同款產品會撞號。
- 條碼作為「品項主檔」的鍵；每一筆檢驗另給**自動流水號 + 時間戳**作為主鍵，再以條碼當外鍵關聯品項。
- 同一條碼會對應**多筆**檢驗記錄（一對多），這正是統計不良率的基礎。

> 若未來需要追溯到「單一件」（哪一件不良），EAN-13 無法提供，需額外引入內部批號 / 序號（見第 11 節）。

### 2.2 人機協作
延續既定原則：**自動化負責掃描與解碼，人保留最終判定權**。NG/OK 這個判定刻意由人按下，系統不自動判良劣，只負責忠實記錄與事後分析。

## 3. 系統架構

```
┌─────────────────────────┐
│  即時辨識系統 (已存在)    │  解出 barcode 數字
└───────────┬─────────────┘
            │ 呼叫 record_inspection(barcode, "OK"/"NG")
            ▼
┌─────────────────────────┐
│  資料存取層 (db.py)       │  ← 共用模組，唯一碰資料庫的地方
│  - 品項自動建檔           │
│  - 寫入檢驗記錄           │
│  - 查詢 / 統計函式        │
└───────────┬─────────────┘
            │
            ▼
      ┌───────────┐        ┌──────────────────────────┐
      │ SQLite .db │◀──────│  後台 UI (Streamlit)       │
      └───────────┘  查詢   │  - 即時記錄頁              │
                            │  - 歷史查詢                │
                            │  - 統計儀表板 (不良率)     │
                            │  - 匯出 CSV / Excel        │
                            └──────────────────────────┘
```

**三層解耦**：辨識、資料存取、UI 各自獨立。日後要換 UI、升級資料庫或加報表，都不會動到辨識端。所有對資料庫的操作集中在 `db.py`，是整個系統唯一的「真相來源」。

## 4. 資料庫 Schema

### 4.1 資料表設計

**`products`（品項主檔）**

| 欄位 | 型別 | 說明 |
|------|------|------|
| `barcode` | TEXT PK | EAN-13 等條碼數字，品項唯一鍵 |
| `name` | TEXT | 品名（可選，首次掃到可留空後補） |
| `spec` | TEXT | 規格備註（可選） |
| `created_at` | TEXT | 建檔時間（ISO8601） |

**`inspections`（檢驗記錄）**

| 欄位 | 型別 | 說明 |
|------|------|------|
| `id` | INTEGER PK AUTOINCREMENT | 流水號，每筆檢驗唯一 |
| `barcode` | TEXT FK→products | 關聯品項 |
| `result` | TEXT | 判定：`OK` / `NG` |
| `inspected_at` | TEXT | 檢驗時間（ISO8601，預設現在） |
| `note` | TEXT | 備註（可選，NG 原因等） |

### 4.2 建表 SQL

```sql
CREATE TABLE IF NOT EXISTS products (
    barcode    TEXT PRIMARY KEY,
    name       TEXT,
    spec       TEXT,
    created_at TEXT NOT NULL DEFAULT (datetime('now','localtime'))
);

CREATE TABLE IF NOT EXISTS inspections (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    barcode      TEXT NOT NULL,
    result       TEXT NOT NULL CHECK (result IN ('OK','NG')),
    inspected_at TEXT NOT NULL DEFAULT (datetime('now','localtime')),
    note         TEXT,
    FOREIGN KEY (barcode) REFERENCES products(barcode)
);

-- 加速「依品項查歷史」與「依時間查當日」
CREATE INDEX IF NOT EXISTS idx_insp_barcode ON inspections(barcode);
CREATE INDEX IF NOT EXISTS idx_insp_time    ON inspections(inspected_at);
```

> `result` 用 `CHECK` 限制只能是 OK/NG，從資料庫層擋掉髒資料。

## 5. 技術選型

| 項目 | 選擇 | 理由 |
|------|------|------|
| 資料庫 | **SQLite** | 單機個人零部署，一個 `.db` 檔搞定，Python 內建支援 |
| 資料存取 | `sqlite3`（內建）或 **SQLAlchemy** | 內建即可起步；要型別安全/未來換 PostgreSQL 再上 SQLAlchemy |
| 後台 UI | **Streamlit** | 單機跑、純 Python 寫網頁介面，最快做出查詢與儀表板 |
| 圖表 | Streamlit 內建 / Plotly | 不良率趨勢、缺陷 Pareto |
| 匯出 | pandas → CSV / Excel | 一行 `to_csv` / `to_excel` |

> 起步建議用內建 `sqlite3` 把 `db.py` 寫乾淨；當你決定未來要上多人/雲端（PostgreSQL），再把 `db.py` 內部換成 SQLAlchemy，UI 與辨識端完全不用改——這正是第 3 節「集中存取」的好處。

## 8. 後台功能（Streamlit）

| 分頁 | 功能 |
|------|------|
| **① 即時記錄** | 顯示辨識端最新掃入的條碼與判定，可手動補打 barcode + 按 OK / NG |
| **② 歷史查詢** | 依條碼 / 日期區間 / 判定篩選，表格列出，支援匯出 CSV、Excel |
| **③ 統計儀表板** | 全廠 / 單品項不良率、每日趨勢折線、NG 品項排行（Pareto） |
| **④ 品項主檔** | 維護品名、規格；補齊只有條碼沒品名的項目 |

## 9. 與「即時辨識系統」的整合

最乾淨且最適合單機的方式：**共用 `db.py`，辨識端解碼後直接呼叫函式**，不需另起 API 服務。

```python
# 即時辨識端：解出條碼後
from db import init_db, record_inspection

init_db()  # 啟動時呼叫一次

# ...辨識迴圈中，當人按下 OK / NG 鈕：
barcode = decode_result          # 你的辨識系統解出的數字
insp_id = record_inspection(barcode, "NG", note="外觀刮傷")
print(f"已記錄 #{insp_id}")
```

辨識端與後台共讀同一個 `qc.db`。SQLite 支援多進程讀寫（同機），個人使用情境下足夠。

> 若未來辨識端與後台要分機運行，再把 `db.py` 包成一個 FastAPI 服務、辨識端改用 HTTP POST 即可——介面（第 7 節函式）不變。
