# 0.py 相機焦距測試工具改進方向

## 文件目的

本文件整理 `0.py` 的現況、問題、改進目標、建議路線、分階段工作內容與驗證標準。

這份文件的定位是「方向確認文件」，不是正式工作清單。若本方向確定，後續應再依 `Collaboration_Standards` 將實作項目拆成 `codex_to_do/0706_1_list*.md`。

目前尚未修改 `0.py`。本文件只負責把後續改進方向寫清楚，方便決定下一步要不要進入實作。

## 一句話結論

`0.py` 應先定位成「獨立的相機手動焦距可行性測試工具」，短期不要急著接進正式 GUI 或 Web 系統。建議先完成穩定性整理與可量化焦距測試，確認相機與驅動真的支援手動焦距控制後，再規劃正式校正流程。

## 目前程式定位

`0.py` 目前是一支 OpenCV 測試腳本，主要用途是確認 Windows 環境下的相機是否能透過程式控制手動焦距。

它不屬於目前正式檢測流程的核心模組，也沒有被 `main.py`、`valve_gui` 或 `valve_web` 匯入使用。因此後續改動應先保持獨立，避免過早影響桌面 GUI、Web UI、相機管理、推論流程或品管資料庫。

## 目前功能摘要

目前 `0.py` 的流程如下：

1. 使用 `cv2.VideoCapture(0, cv2.CAP_DSHOW)` 嘗試開啟第 0 號相機。
2. 如果 DirectShow 失敗，改用 `cv2.VideoCapture(0)` 再試一次。
3. 相機仍無法開啟時，輸出錯誤並結束。
4. 呼叫 `cap.set(cv2.CAP_PROP_AUTOFOCUS, 0)` 嘗試關閉自動對焦。
5. 依序測試焦距值 `0, 60, 120, 180, 240`。
6. 每個焦距值讀取 15 幀畫面，並在畫面上顯示目前測試焦距。
7. 每組焦距後暫停 1 秒，讓使用者用肉眼觀察清晰度。
8. 最後呼叫 `cap.set(cv2.CAP_PROP_SETTINGS, 1)` 嘗試打開 Windows 相機設定視窗。
9. 正常流程結束時釋放相機並關閉 OpenCV 視窗。

## 明確假設

後續方向基於以下假設：

1. 目前目標是測試相機焦距控制能力，不是直接改主檢測系統。
2. 使用環境主要是 Windows。
3. 相機可能是 UVC 相機，但是否支援程式控制焦距取決於硬體與驅動。
4. 螺絲檢測需要固定、可重複的焦距設定，不能長期依賴自動對焦。
5. `0.py` 可以先作為開發測試工具保留在專案根目錄，等流程成熟後再決定是否移入正式模組。
6. 目前不應改動 SQLite、GUI、Web API 或正式相機管理流程，除非焦距控制已經被實機驗證有效。

## 現況判斷

目前 `0.py` 沒有明顯語法錯誤，中文註解與輸出也能閱讀。它已經可以作為最基本的焦距測試腳本。

真正需要改進的不是「讓它能跑」，而是：

1. 讓它可以安全中止。
2. 讓相機資源一定會被釋放。
3. 讓測試結果可比較、可記錄。
4. 讓使用者知道相機到底是支援、部分支援，還是完全不支援手動焦距。
5. 為未來正式導入焦距校正流程留下清楚路線。

## 核心問題

### 1. 無法提前中止測試

目前程式有呼叫 `cv2.waitKey(30)`，但沒有判斷按鍵。

影響：

- 使用者按 `q` 或 `Esc` 不會中止流程。
- 畫面異常、焦距已確認、相機卡住時，只能等測試全部跑完。
- 實機測試體驗不好。

明確改進方向：

- 在每幀預覽時讀取按鍵。
- 支援 `q` 與 `Esc` 提前離開。
- 中止時要跳出外層焦距測試，不只跳出內層幀迴圈。

### 2. 相機釋放不夠保險

目前只有正常流程結尾會執行：

```python
cap.release()
cv2.destroyAllWindows()
```

如果中途發生例外，可能無法執行釋放。

影響：

- 相機可能被占用。
- 下一次執行 `0.py` 或正式 GUI 時，相機可能開不起來。
- 使用者會誤判為相機或主程式故障。

明確改進方向：

- 將主要流程包進 `main()`。
- 使用 `try/finally` 統一釋放相機與視窗。
- `cap` 建立失敗時也要避免後續呼叫未初始化物件。

### 3. `cap.set()` 回傳成功不代表鏡頭真的移動

OpenCV 的 `cap.set(cv2.CAP_PROP_FOCUS, value)` 回傳 `True`，只能代表驅動接受請求，不保證實際焦距有變。

常見情況：

- 相機硬體不支援手動焦距。
- 驅動忽略焦距設定。
- 焦距值範圍不是 0 到 255。
- 焦距值只能接受特定步進。
- 必須先從驅動控制面板關閉自動對焦。
- `cap.get(cv2.CAP_PROP_FOCUS)` 回讀值不可靠。

明確改進方向：

- 每次輸出「要求焦距值」、「設定回傳值」、「實際回讀值」。
- 不把 `set=True` 當成焦距有效的唯一判斷。
- 加入清晰度分數，透過影像變化輔助判斷焦距是否真的生效。

### 4. 目前固定焦距值只能做粗略確認

目前測試值是：

```python
focus_values = [0, 60, 120, 180, 240]
```

這適合快速看鏡頭是否有反應，但不適合找最佳焦距。

影響：

- 最佳焦距可能落在 60 和 120 之間。
- 若未來用於螺絲檢測，焦距不準會影響 YOLO 偵測穩定性。
- 用肉眼判斷容易受螢幕大小、光線、操作者經驗影響。

明確改進方向：

- 第一階段保留固定值快速測試。
- 第二階段加入範圍掃描，例如 `0..255` 每 10 或 20 一格。
- 後續若要更準，再針對分數最高附近做細掃。

### 5. 缺少量化清晰度指標

目前只能靠肉眼看畫面變清楚或變模糊。

影響：

- 測試結果不容易重現。
- 不同操作者可能判斷不同。
- 無法自動找出最佳焦距。

明確改進方向：

- 使用 Laplacian variance 計算清晰度分數。
- 每個焦距值讀多幀，取平均分數或中位數。
- 顯示分數最高的焦距值。
- 未來若有螺絲 ROI，應優先計算 ROI 的清晰度，不是整張畫面。

### 6. 測試結果沒有保存

目前測試結果只顯示在終端機與畫面上。

影響：

- 每次測試後需要人工記錄。
- 換相機、換位置、重開機後不容易追蹤差異。
- 無法作為正式校正流程的資料來源。

明確改進方向：

- 第二階段可先輸出文字摘要。
- 第三階段再考慮存成 JSON 或 CSV。
- 只有在確認要整合正式系統時，才寫入 `app_config.json`。

## 建議採用路線

建議採用「A 到 B，再評估 C」的路線。

### 階段 A：整理成穩定測試工具

這是最優先、最小範圍的改動。

目標：

- 保持 `0.py` 是獨立腳本。
- 不改 GUI。
- 不改 Web。
- 不改正式相機管理。
- 只讓測試工具更安全、更清楚。

應做內容：

1. 建立 `main()`。
2. 建立 `open_camera(index)`。
3. 支援 `q` / `Esc` 中止。
4. 使用 `try/finally` 釋放相機。
5. 統一輸出格式。
6. 保留 `cv2.CAP_PROP_SETTINGS` 作為手動確認入口。

不做內容：

- 不自動寫入正式設定檔。
- 不接入 GUI。
- 不接入 Web。
- 不加入複雜參數解析。
- 不建立新套件結構。

完成標準：

1. 執行 `python 0.py` 可開啟相機或清楚說明失敗原因。
2. 測試過程按 `q` 或 `Esc` 可離開。
3. 離開後相機與 OpenCV 視窗確實釋放。
4. 再次執行 `python 0.py` 不會因上一輪未釋放而失敗。
5. 終端機能看出每個焦距值的設定結果與回讀值。

### 階段 B：加入清晰度分數與最佳焦距建議

階段 A 完成後再進行。

目標：

- 不再只靠肉眼判斷。
- 讓每個焦距值有量化分數。
- 找出目前畫面下最清晰的焦距值。

應做內容：

1. 新增 `sharpness_score(frame)`。
2. 使用灰階 Laplacian variance 計算清晰度。
3. 每個焦距值讀多幀，略過前幾幀等待鏡頭穩定。
4. 每個焦距輸出平均清晰度分數。
5. 測試結束後列出最佳焦距候選。

建議測試策略：

1. 快速測試：`[0, 60, 120, 180, 240]`。
2. 粗掃測試：`range(0, 256, 20)`。
3. 細掃測試：在最佳粗掃值前後 `±20` 以每 5 一格測。

完成標準：

1. 清晰畫面的分數明顯高於模糊畫面。
2. 相同焦距重複測試時，分數大致穩定。
3. 程式能輸出分數最高的焦距值。
4. 若不同焦距分數幾乎不變，文件或輸出要提示「可能不支援手動焦距或畫面內容不適合判斷」。

### 階段 C：正式焦距校正流程

只有在階段 A、B 都通過實機驗證後，才進入階段 C。

目標：

- 把焦距控制從測試工具提升為正式檢測站校正流程。
- 讓每台相機都有可保存、可重複套用的焦距設定。

可能工作內容：

1. 在相機設定資料中加入 focus 設定欄位。
2. 在 GUI 設定頁加入手動焦距設定或校正按鈕。
3. 在 Web 設定頁加入對應控制。
4. 在 `VideoSource` 或相機初始化流程中套用焦距。
5. 將每台 Camera slot 的最佳焦距寫入 `app_config.json`。
6. 若已有 ROI，校正時計算 ROI 清晰度，而不是整張畫面。

進入條件：

1. 實機確認相機可由程式關閉自動對焦。
2. 至少兩個不同焦距值能造成明顯清晰度差異。
3. 清晰度分數能穩定選出合理焦距。
4. 現場檢測距離、光源與相機固定方式已大致確定。
5. 團隊確認焦距設定要納入正式操作流程。

不建議現在直接做階段 C。

原因：

- 目前仍不知道相機與驅動是否真的可靠支援手動焦距。
- 過早改主系統會增加 GUI、Web、設定檔與相機管理的連動風險。
- 若硬體不支援，主系統整合會變成無效功能。

## 最終建議方向

目前最明確的方向是：

1. 先把 `0.py` 修成安全、可中止、輸出清楚的測試工具。
2. 實機測一次，確認相機是否能被程式控制焦距。
3. 若焦距確實有反應，再加入清晰度分數與最佳焦距建議。
4. 若清晰度分數也穩定，再討論是否整合到正式 GUI / Web。

換句話說，現在不要直接做「正式焦距校正系統」。現在要先做「焦距控制能力驗證工具」。

def open_camera(index: int):
    ...

def set_manual_focus(cap, value: int):

    ...

各函式責任：

4. `open_driver_settings(cap)`
   - 測試結束後開啟驅動設定視窗。

5. `main()`
   - 管理流程。
   - 使用 `try/finally` 確保釋放資源。

## 階段 B 詳細實作方向

建議新增以下函式：

```python
def sharpness_score(frame) -> float:
    gray = cv2.cvtColor(frame, cv2.COLOR_BGR2GRAY)
    return cv2.Laplacian(gray, cv2.CV_64F).var()
```

測試流程：

1. 設定焦距。
2. 先讀取幾幀但不計分，等待畫面穩定。
3. 再讀取多幀計算清晰度。
4. 對每個焦距輸出平均分數。
5. 最後排序並列出最佳焦距。

建議輸出格式：

```text
Focus 120 | set=True | readback=120.0 | sharpness=845.32
Best focus candidate: 120 (sharpness=845.32)
```

若分數差異很小，應提示：

```text
清晰度分數差異很小，可能是焦距控制未生效，或畫面內容不適合用整張圖判斷。
```

## 實機驗證流程

### 基本驗證

1. 執行：

```powershell
python 0.py
```

2. 確認相機畫面出現。
3. 確認每個焦距值都有終端機輸出。
4. 按 `q` 或 `Esc` 能提前結束。
5. 結束後再次執行，確認相機沒有被占用。

### 焦距有效性驗證

1. 把測試物固定在螺絲檢測實際距離。
2. 固定光源，不要手持相機。
3. 觀察焦距值變化時畫面是否真的清楚/模糊。
4. 比對 `set()`、`get()` 與清晰度分數。

### 判定結果

可分成三種情況：

1. 明確支援
   - 焦距值改變時畫面明顯變化。
   - 清晰度分數也跟著變化。
   - 可進入階段 B 或 C。

2. 部分支援
   - 某些值有效，某些值被忽略。
   - 需要找出可用範圍與步進。
   - 先不要整合主系統。

3. 不支援
   - 焦距值改變但畫面與分數都不變。
   - 回讀值固定或不可信。
   - 應改用驅動工具手動固定焦距，或更換支援手動焦距的相機。

## 正式整合前需確認

若未來要把焦距設定接進正式系統，至少需要先確認：

1. 每台檢測相機的型號是否固定。
2. 每台相機的焦距值範圍是否一致。
3. 相機是否每次重開都需要重新關閉自動對焦。
4. `cv2.CAP_PROP_AUTOFOCUS` 在目前相機上是否有效。
5. `cv2.CAP_PROP_FOCUS` 是否可在 `VideoSource` 初始化後立即設定。
6. 是否需要每個 Camera slot 存一組焦距。
7. 焦距校正是給開發者使用，還是管理者也能操作。
8. GUI 與 Web 是否都要支援同一套設定。
9. 最佳焦距是否要和 ROI 綁定。
10. 焦距設定變更是否需要留下操作紀錄。

## 不建議做的事

目前不建議：

1. 直接把 `0.py` 接進 `main.py`。
2. 直接修改 `valve_gui.camera.VideoSource` 加焦距設定。
3. 直接修改 `app_config.json` schema。
4. 直接在 GUI 或 Web 增加焦距設定頁。
5. 在沒有實機驗證前承諾自動對焦校正可用。
6. 把清晰度分數當成 YOLO 檢測準確率。

理由是焦距控制高度依賴硬體與驅動。沒有先確認硬體支援，主系統整合會增加維護成本但不一定有實際效果。

## 風險與對策

| 風險 | 影響 | 對策 |
| --- | --- | --- |
| 相機不支援手動焦距 | 程式設定無效 | 先用 `0.py` 驗證，不急著整合主系統 |
| 回讀焦距值不可信 | 誤判設定成功 | 同時看畫面與清晰度分數 |
| 自動對焦無法關閉 | 焦距會自動漂移 | 嘗試驅動設定視窗，必要時更換相機 |
| 光線不穩 | 清晰度分數波動 | 固定光源與測試物距離 |
| 整張畫面干擾太多 | 分數不能代表螺絲清晰度 | 後續改用螺絲 ROI 計分 |
| 相機未釋放 | 正式程式開不起相機 | `try/finally` 強制釋放 |

## 成功標準

短期成功標準：

1. `0.py` 可安全啟動與結束。
2. 使用者可按 `q` 或 `Esc` 中止。
3. 相機資源不會殘留占用。
4. 終端機輸出足以判斷每個焦距設定是否被接受。
5. 能初步判斷相機是否支援手動焦距。

中期成功標準：

1. 程式可輸出清晰度分數。
2. 能列出最佳焦距候選。
3. 重複測試結果大致一致。
4. 能辨識「焦距控制無效」的情況，而不是誤報成功。

長期成功標準：

1. 每台檢測相機可保存固定焦距。
2. 主系統啟動時能自動套用焦距。
3. 焦距校正流程可由指定權限角色操作。
4. 螺絲 ROI 的清晰度可作為校正依據。
5. 焦距設定不會破壞現有 GUI、Web、推論與歷史紀錄流程。

## 建議下一步
