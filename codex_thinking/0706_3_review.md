# 鎖緊 / 分開狀態判斷方案規劃

## 文件目的

本文件整理「同一位置的零件狀態判斷」方案，來源是兩張實際畫面：

```text
Image #1：鎖緊
Image #2：分開
```

使用者希望判斷的不是「有沒有物件」，而是「該位置是否鎖緊 / 是否分開」。

本文件也回答一個關鍵問題：

```text
有沒有現成的？
```

## 一句話結論

沒有一個可以直接拿來判斷你這個零件「鎖緊 / 分開」的現成模型，因為這是特定產品、特定相機角度、特定治具位置的視覺狀態判斷。

但是有現成的方法與框架可以用：

1. OpenCV 規則法。
2. YOLO 偵測多類別法。
3. YOLO / CNN ROI 二分類法。
4. 工業異常檢測法。

以目前專案狀態來看，最建議先做：

```text
固定 ROI 裁切
→ 訓練二分類模型：locked / separated
→ 把分類結果接回目前 PASS / NG 判定流程
```

## 目前專案現況

目前專案文件中有定義過：

```text
screw_ok
screw_missing
screw_not_locked
```

位置：

```text
docs/annotation_spec.md
training/data/yolo_dataset/dataset.yaml
```

但目前實際使用的三個模型權重不是這種多類別狀態模型。

目前 `models/` 底下三個權重的 class names 是：

```text
Screw_baseline20260507/best.pt  -> 0: screw
Label_baseline20260507/best.pt  -> 0: label_ok
Filter_baseline20260507/best.pt -> 0: filter_ok
```

也就是說：

1. 螺絲模型目前只知道 `screw`。
2. 標籤模型目前只知道 `label_ok`。
3. 濾網模型目前只知道 `filter_ok`。

它們目前不能直接判斷：

```text
locked
separated
screw_ok
screw_not_locked
```

目前系統能做的是：

```text
偵測到幾個框
→ 根據數量與信心值判斷 PASS / NG
```

但它還不能穩定判斷兩個零件的貼合狀態。

## 人工判斷依據

從使用者提供的兩張圖來看，人工判斷不是看「有沒有金屬件」，而是看以下特徵：

### 1. 接觸間隙

鎖緊狀態：

```text
零件貼合定位面，沒有明顯分離縫。
```

分開狀態：

```text
零件與座面之間有明顯空隙或黑色縫隙。
```

### 2. 零件相對位置

鎖緊狀態：

```text
金屬件落在固定位置，位置穩定且貼近基準面。
```

分開狀態：

```text
金屬件偏離固定位置，看起來懸空、浮起或未壓到底。
```

### 3. 陰影與亮暗變化

鎖緊狀態：

```text
陰影主要集中在零件邊緣。
```

分開狀態：

```text
分離處會出現較明顯的暗縫或陰影區。
```

### 4. 固定 ROI 內的形狀差異

這類問題通常不是全圖判斷，而是某一個固定小區域判斷。

因此模型或規則應該聚焦在：

```text
該鎖緊位置的 ROI
```

而不是整張相機畫面。

## 有沒有現成的？

### 1. 現成模型：沒有

如果問題是：

```text
有沒有一個已訓練好的模型，可以直接判斷我的產品這個位置是鎖緊還是分開？
```

答案是：

```text
通常沒有。
```

原因：

1. 你的零件外觀是特定的。
2. 相機角度是特定的。
3. 光源與治具是特定的。
4. 鎖緊 / 分開的差異很細微。
5. 通用模型沒有看過你的產品狀態。

現成通用模型最多能幫忙偵測常見物件，但不會知道你這個產品「這樣算鎖緊、那樣算分開」。

### 2. 現成方法：有

雖然沒有直接可用的模型，但有成熟方法可以套用。

可選方案：

```text
A. OpenCV 規則法
B. YOLO 多類別偵測
C. ROI 二分類模型
D. 工業異常檢測模型
```

## 方案 A：OpenCV 規則法

### 概念

相機固定、位置固定、光源固定時，可以先裁切 ROI，然後用影像規則判斷。

可能規則：

1. 黑色縫隙面積。
2. 金屬件邊緣位置。
3. 亮暗比例。
4. 與鎖緊樣板的差異。
5. 邊緣線距離。

可能使用：

```text
threshold
edge detection
template matching
contour area
pixel intensity profile
```

### 優點

1. 不需要大量資料。
2. 可以很快做 MVP。
3. 計算速度快。
4. 結果可解釋。

### 缺點

1. 光線變化會影響結果。
2. 相機位置稍微偏移就可能失準。
3. 零件反光會造成誤判。
4. 規則需要現場調參。
5. 泛化能力較差。

### 適合情境

如果現場條件穩定：

```text
相機固定
治具固定
光源固定
產品位置固定
```

可以先用這個做最快 MVP。

## 方案 B：YOLO 多類別偵測

### 概念

重新訓練一個 YOLO 模型，讓它直接偵測不同狀態：

```text
locked
separated
```

或沿用文件命名：

```text
screw_ok
screw_not_locked
```

### 優點

1. 可直接在畫面上顯示狀態框。
2. 與目前系統 YOLO 流程最接近。
3. 可以接進現有 `InferenceRouter`。
4. 結果直觀。

### 缺點

1. 差異很小時，YOLO 偵測框不一定是最佳方法。
2. 如果兩種狀態位置非常接近，框的位置可能差不多，分類才是關鍵。
3. 需要重新標註資料。
4. 需要足夠的鎖緊與分開樣本。

### 適合情境

如果除了鎖緊狀態，還需要同時定位物件位置，可以考慮 YOLO 多類別偵測。

## 方案 C：ROI 二分類模型

### 概念

先用固定 ROI 裁切該位置，再訓練分類模型判斷：

```text
locked
separated
```

輸入：

```text
小張 ROI 圖片
```

輸出：

```text
locked / separated
confidence
```

### 優點

1. 最符合這次問題本質。
2. 不需要模型學整張畫面。
3. 對「細微位置差異」通常比偵測框更合理。
4. 標註成本低，只需要圖片資料夾分類。
5. 很適合固定相機、固定治具的工業檢測。

### 缺點

1. 需要收集兩類圖片。
2. ROI 位置必須穩定。
3. 若產品位置漂移，需先定位或做較大 ROI。

### 建議類別

建議先用英文 class name：

```text
locked
separated
```

或如果要和現有文件一致：

```text
screw_ok
screw_not_locked
```

我會建議第一版用：

```text
locked
separated
```

原因是這次判斷看起來更像「零件貼合狀態」，不一定完全等於螺絲本身。

## 方案 D：工業異常檢測

### 概念

只收集大量正常鎖緊圖片，模型學會正常狀態，未鎖緊或分開就判定為異常。

常見概念：

```text
normal vs anomaly
```

### 優點

1. 異常樣本少時也可嘗試。
2. 適合缺陷種類很多、不容易全部標註時。
3. 可以找出與正常樣板差異大的位置。

### 缺點

1. 對部署與調參要求較高。
2. 輸出通常不是直接的 locked / separated。
3. 對光線、反光、位置漂移敏感。
4. 目前專案架構還沒有異常檢測流程。

### 適合情境

如果未來缺陷種類很多，例如：

```text
分開
歪斜
偏移
缺件
變形
反裝
```

可以再考慮異常檢測。

## 四種方案比較

| 方案 | 是否現成 | 資料需求 | 開發速度 | 穩定性 | 建議程度 |
| --- | --- | --- | --- | --- | --- |
| OpenCV 規則法 | 方法現成 | 低 | 快 | 中等，依現場而定 | MVP 可用 |
| YOLO 多類別 | 框架現成，模型需訓練 | 中 | 中 | 中高 | 可選 |
| ROI 二分類 | 框架現成，模型需訓練 | 中低 | 中 | 高 | 最建議 |
| 異常檢測 | 框架現成，需導入 | 中 | 慢 | 中 | 後續再看 |

## 我建議的方向

最建議：

```text
第一階段：OpenCV 規則 MVP
第二階段：ROI 二分類模型
第三階段：接入正式 PASS / NG 判定
```

原因：

1. OpenCV 規則可以快速驗證 ROI 是否足以判斷。
2. 如果規則可以分開兩類，就表示這個 ROI 有足夠視覺差異。
3. 接著用 ROI 二分類模型會更穩。
4. 不需要一開始就重訓完整 YOLO。

## 第一階段 MVP：OpenCV 規則法

### 目標

先不用訓練模型，只用固定 ROI 判斷兩張圖差異。

### 做法

1. 在監視畫面設定 ROI。
2. 裁切鎖緊位置。
3. 轉灰階。
4. 針對分離縫隙位置計算暗色區域比例。
5. 設定 threshold。
6. 輸出 locked / separated。

### 判斷概念

```text
暗縫比例小 → locked
暗縫比例大 → separated
```

或：

```text
金屬件邊緣位置在正常範圍內 → locked
金屬件邊緣偏離正常範圍 → separated
```

### 驗證資料

至少先收集：

```text
locked：20 張
separated：20 張
```

如果 40 張能分得開，再繼續。

## 第二階段：ROI 二分類模型

### 資料夾建議

```text
training/data/lock_state/
├─ locked/
│  ├─ locked_0001.jpg
│  ├─ locked_0002.jpg
│  └─ ...
└─ separated/
   ├─ separated_0001.jpg
   ├─ separated_0002.jpg
   └─ ...
```

### 圖片來源

每張圖應該是裁切後 ROI，而不是整張相機畫面。

建議每類至少：

```text
50 到 100 張
```

更穩定的版本：

```text
每類 200 張以上
```

### 資料要包含變化

每類都要包含：

1. 光線微變化。
2. 相機微小抖動。
3. 產品正常位置微差。
4. 不同批次零件。
5. 輕微反光。
6. 清晰與稍微模糊的可接受畫面。

不要只拍同一張角度，否則模型只會背答案。

## 第三階段：接入正式系統

### 建議整合點

目前系統已有：

```text
CameraConfig.detection_regions
CameraConfig.region_detection_enabled
InferenceRouter
DecisionConfig
```

可以沿用 ROI 設定概念。

未來可以新增：

```text
Classifier model
ROI crop
classification result
```

判定流程：

```text
相機取得 frame
→ 套用旋轉 / 翻轉
→ 裁切指定 ROI
→ 分類 locked / separated
→ locked = PASS 條件之一
→ separated = NG
```

### 第一版整合方式

先不改很多架構，可以做成一個新推論類型：

```text
model.modality = "classifier"
```

目前 `SettingsPage` 裡其實已經有：

```python
MODEL_MODALITIES = ["vision", "text", "multimodal", "ocr", "classifier"]
```

但目前 `InferenceRouter` 還沒有真正支援 classifier。

因此後續若要正式接入，需要補：

1. classifier 模型載入。
2. ROI crop。
3. classifier 推論。
4. classifier 結果轉成 PASS / NG。

## 是否要重新訓練 YOLO？

不一定。

如果目標只是判斷這個固定位置：

```text
locked / separated
```

我會優先做 ROI 分類，不會先重訓 YOLO。

如果目標變成：

```text
找出畫面中所有可能未鎖緊位置
```

或相機/產品位置不固定，才更需要 YOLO 偵測。

## 最小可行資料量

### OpenCV MVP

```text
locked：20 張
separated：20 張
```

用途：

```text
驗證特徵是否分得開
```

### ROI 二分類初版

```text
locked：50 到 100 張
separated：50 到 100 張
```

用途：

```text
訓練初版分類器
```

### 比較穩的現場版

```text
locked：200 張以上
separated：200 張以上
```

用途：

```text
降低光線、反光、批次差異造成的誤判
```

## 標註方式

若採 ROI 二分類，不需要畫 bounding box。

只需要把圖片放到對應資料夾：

```text
locked/
separated/
```

這比 YOLO box 標註快很多。

## 對現有系統的影響

短期不需要改：

1. YOLO 螺絲模型。
2. 標籤模型。
3. 濾網模型。
4. SQLite 品管資料庫。
5. 歷史紀錄。

第一步只需要做資料收集與離線驗證。

真正接入系統時，才需要改：

1. `InferenceRouter`
2. `ModelConfig`
3. ROI crop 邏輯
4. 判定規則
5. 設定頁模型類型

## 風險

| 風險 | 影響 | 對策 |
| --- | --- | --- |
| 光線反光太強 | 規則法與模型都可能誤判 | 固定光源，加遮光，收集反光樣本 |
| ROI 太小 | 分類資訊不足 | ROI 包含零件、接觸面、分離縫 |
| ROI 太大 | 背景干擾 | 先固定 ROI，只包含必要區域 |
| 樣本太少 | 模型背答案 | 每類至少 50 到 100 張 |
| 鎖緊/分開界線不清楚 | 標註不一致 | 先定義人工判斷規則 |
| 相機焦距不穩 | 圖像差異變大 | 先完成焦距固定功能 |

## 建議下一步

建議先不要直接訓練模型，也不要先改主系統。

下一步應該是：

1. 固定相機、光源、治具。
2. 針對該鎖緊位置裁切 ROI。
3. 收集 `locked` 與 `separated` 各 20 張。
4. 用 OpenCV 簡單計算暗縫或位置差異。
5. 如果能分開，再建立分類資料集。
6. 若分類資料集穩定，再規劃接入 `InferenceRouter`。

## 回答：有現成的嗎？

結論：

```text
沒有現成模型可以直接判斷你的零件鎖緊 / 分開。
但有現成方法可以快速做。
```

最實際的選擇：

```text
短期：OpenCV ROI 規則
中期：ROI 二分類模型
長期：整合 classifier 到目前系統
```

如果目標是最快看到結果，我建議先做 OpenCV 規則 MVP。

如果目標是正式穩定上線，我建議收集 ROI 圖片後訓練二分類模型。
