## 現有架構重點

目前檢測流程大致是：

```text
live frame
  -> InferenceRouter.run(frames)
  -> InferenceResult
  -> InspectionRecord
  -> CSV / SQLite / qc_objects
```

重要檔案：

- `app/src/valve_gui/pages/monitor.py`
- `app/src/valve_gui/inference_router.py`
- `app/src/valve_gui/main_window.py`
- `app/src/valve_gui/models.py`
- `app/src/valve_gui/storage.py`
- `app/src/valve_gui/qc_db.py`
- Web 端若第二階段要同步，才看 `app/src/valve_web/routers/inspect.py`

目前桌面端單次檢測是直接從 `last_frames` 取 frame 後推論；尚未有完整的掃碼、倒數、固定快照交易流程。

## 第一階段目標

第一階段只做桌面端主流程雛形：

```text
等待掃碼
  -> 掃碼完成
  -> 倒數
  -> 倒數結束拍照
  -> 使用固定快照推論
  -> 顯示結果
  -> 交給現有 add_record 保存
```

成功標準：

1. 檢測開始後會建立一個本次交易狀態。
2. 倒數結束時會 copy 當下 active camera frames。
3. `InferenceRouter.run()` 使用這組固定 frames，不在推論中重新抓 live frame。
4. 推論完成後，結果畫面與保存流程使用同一筆 inference result。
5. 下一件開始前，前一件 RAM frame 不會被覆蓋到尚未完成的交易。

## 要做的工作

### 1. 復盤現有桌面端檢測入口

檢查：

- `MonitorPage.inspect_once()`
- `MonitorPage.detect_current_frames()`
- `MonitorPage.apply_detection_result()`
- `MonitorPage.record_detection()`
- `MainWindow.add_record()`

確認哪些地方目前直接使用 `last_frames`，哪些地方會寫入紀錄。

### 2. 建立桌面端交易狀態

在 `MonitorPage` 加入目前交易物件或等價狀態。

至少要能回答：

- 目前是否正在處理一件產品
- 目前流程在哪個 step
- 這件產品的條碼是什麼
- 倒數結束時拍到哪些 raw frames
- 推論是否完成
- 是否可進入下一件

### 3. 加入倒數流程

先做最小可用版本即可。

建議第一版先支援「按確認後倒數」：

```text
使用者確認檢測
  -> 倒數 1~2 秒
  -> capture frames
  -> run inference
```

如果要同時支援「掃碼完成自動倒數」，請做成設定值或清楚的常數，不要把兩套流程寫死混在一起。

### 4. 倒數結束後固定快照

倒數結束當下：

```python
frames = {slot: frame.copy() for slot, frame in self.last_frames.items()}
```

把這組 frames 放進 transaction，後續都從 transaction 取。

注意：

- 不要直接把 `last_frames` dict 指標交出去。
- 每台相機 frame 要 copy。
- active camera 沒有 frame 時，要進入 error state 或提示，不能直接推論不完整資料。

### 5. 讓推論只吃 transaction frames

現有 `InferenceRouter.run(frames_by_slot)` 可以沿用。

要改的是呼叫端：

```text
transaction.raw_frames
  -> self.router.run(transaction.raw_frames)
```

推論完成後，把 `InferenceResult` 放回 transaction。

### 6. 保留現有保存流程

第一階段不要重寫 SQLite / CSV / qc_objects。

推論完成後仍走：

```text
apply_detection_result()
record_detection()
MainWindow.add_record()
```

但要確認保存用到的是 transaction 的 inference result，而不是推論完成時新的 live frame。

### 7. 補最小錯誤提示

第一階段至少處理：

- 尚未登入不能開始
- 設定未套用不能開始
- 沒有 active camera frame 不能開始
- 倒數中不能重複開始
- 推論中不能進入下一件

錯誤處理不需要一次做到完整 log 系統，但不能靜默失敗。

## 第二階段預留工作

第一階段完成後，再開新工作單處理：

1. 主條碼 / 副條碼資料結構。
2. 掃碼完成自動倒數模式。
3. 照片保存策略：只存 NG、只存 PASS、都存、都不存。
4. CSV 補 `barcode_source` 或 metadata。
5. 結果 Review 畫面完整化。
6. SQLite / CSV / qc_objects 保存失敗提示。
7. Web 端同步桌面端交易流程。
8. 相機 FPS 與背景擷取 thread 優化。

## 驗收方式

下一位 Codex 做完第一階段後，至少要驗證：

1. 單次檢測可正常跑完。
2. 倒數期間不能重複開始檢測。
3. 倒數結束後使用固定 frames 推論。
4. 推論中 live preview 改變，不會影響該次推論結果。
5. 無相機 frame 時不會寫入錯誤紀錄。
6. PASS / NG 結果仍能寫入現有 CSV / SQLite / qc_objects。
7. `git diff` 只包含第一階段相關檔案，沒有順手重構無關頁面。

如果專案目前沒有自動化測試，請至少提供手動驗證步驟與實際跑過的指令。
